## 开发原则

- 保持最小改动，只修改当前需求涉及的代码和文档。
- 优先遵循现有架构、分层、命名和测试风格。
- 业务链路复杂时，先补齐数据流和边界条件，再写代码。
- 生成文件只在明确需要时更新，并记录对应命令。
- 测试优先选择与改动范围匹配的最小集合，复杂改动再做全量测试。

---

## 禁止事项

- 不主动重构无关遗留代码。
- 不引入新依赖，除非需求方案已说明原因和替代方案。
- 不跳过错误处理、日志和边界条件。
- 不把其他项目的示例需求复制为当前项目需求。

## SpecPilot 使用约定

### 需求编号

- 格式：`REQ-001-short-description`
- 编号从 `001` 开始递增
- 描述使用英文小写和连字符

### 执行模式

| 场景 | 推荐模式 |
|------|----------|
| 新增功能 | 先出方案 |
| 跨模块改动 | 先出方案 |
| 数据库/接口/worker/外部集成变化 | 先出方案 |
| 小 bug 修复 | 直接实施 |
| 文档或测试补充 | 直接实施 |

### 附加文件

| 文件 | 创建条件 |
|------|----------|
| `REQ-xxx-design.md` | 跨模块、新流程、数据迁移、接口变化 |
| `REQ-xxx-tasks.md` | 涉及 3 个以上明确实施步骤 |
